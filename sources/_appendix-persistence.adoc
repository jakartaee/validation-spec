== {spec-name-persistence} and schema generation

{spec-name-persistence} defines integration points with {spec-name-bv} in two areas:
automatic validation upon entity lifecycle events, and derivation of schema metadata
from {spec-name-bv} constraints for DDL generation.
The authoritative specification of this integration is defined by
{spec-name-persistence}; this appendix only provides a summary for reference.

=== Integration

The `ValidatorFactory` instance is supplied by the container in Jakarta EE environments
(via the `jakarta.persistence.validation.factory` property), or bootstrapped by the
persistence provider using `Validation.buildDefaultValidatorFactory()` in Java SE
environments when no factory is explicitly provided.

The persistence provider must use a `TraversableResolver` that only considers attributes
and associations already loaded, and does not cascade into associations, ensuring that
validation does not trigger lazy loading.

=== Automatic validation upon lifecycle events

When automatic validation is enabled, the persistence provider validates managed types
(entities, mapped superclasses, and embeddable classes) immediately after lifecycle
callback methods complete for the applicable events.

By default, constraints from the `jakarta.validation.groups.Default` group are validated.
The target groups for each lifecycle event can be overridden via configuration properties
such as `jakarta.persistence.validation.group.*`.

If validation produces any `ConstraintViolation`, the persistence provider must throw a
`jakarta.validation.ConstraintViolationException` and mark the transaction for rollback.

=== Derivation of schema metadata from {spec-name-bv} constraints

The persistence provider must use {spec-name-bv} constraint metadata to derive
persistent attribute characteristics (such as nullability, column length, and
numeric precision) and database constraints (such as check constraints) for the
purpose of DDL schema generation.
This derivation occurs whenever a {spec-name-bv} provider is present and is
independent of the runtime validation mode.

Only constraints belonging to the `jakarta.validation.groups.Default` group are
considered by default, though providers may allow overriding this.

{spec-name-bv} constraint metadata takes precedence over {spec-name-persistence}
annotation metadata when both are present. When multiple constraints apply to the
same attribute, the most restrictive constraint is applied.

The precise list of recognized built-in constraints, their influence on attribute
metadata, and the resulting DDL effects are defined by {spec-name-persistence}.
